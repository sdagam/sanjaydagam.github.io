Reliability Engineering: Predicting When Things Fail

#74 of 156 May 31, 2025 Group 3 · Jan–Jun 2025

A commercial aircraft has tens of thousands of components, each capable of failure. Yet the overall failure rate of modern commercial aviation is extraordinarily low—less than one fatal accident per ten million flights. Achieving this reliability isn't luck or over-engineering; it is the systematic application of reliability engineering: the mathematical discipline of predicting, measuring, and improving the probability that systems perform their required function for a specified time under stated conditions. The same mathematics that keeps aircraft flying also governs medical devices, nuclear plants, and semiconductor chips.

Failure Rate and the Bathtub Curve

The hazard rate h(t) measures the instantaneous probability of failure given survival to time t. For most engineering components, h(t) follows the bathtub curve: high initial failure rate during infant mortality (manufacturing defects, installation errors), a low constant failure rate during useful life (random failures), and increasing failure rate during wearout (fatigue, corrosion, degradation). The exponential distribution—constant hazard h(t) = λ—models the useful life phase; the Weibull distribution generalizes to increasing (β > 1) or decreasing (β < 1) hazard, fitting all three phases with different parameter values.

Exponential: h(t) = λ, R(t) = e^{−λt} Weibull: h(t) = (β/η)(t/η)^{β−1}, R(t) = e^{−(t/η)^β}

System Reliability: Series and Parallel

Complex systems are analyzed by combining component reliabilities using probability theory. Series systems fail if any component fails: R_system = Π Rᵢ. Parallel (redundant) systems fail only if all components fail: R_system = 1 − Π(1 − Rᵢ). Adding redundancy dramatically improves reliability: if each component has R = 0.99, a series system of 100 components has R ≈ 0.37, while doubling up (each component has a redundant backup) gives R ≈ 0.86. Fault tree analysis (FTA) builds Boolean logic trees of failure combinations; reliability block diagrams map system topology to probability calculations.

Series: R_sys = Π Rᵢ Parallel: R_sys = 1 − Π(1−Rᵢ) Example: 2 components R=0.9 in parallel → R_sys = 1−(0.1)² = 0.99

Failure Mode Analysis

Failure Mode and Effects Analysis (FMEA) systematically identifies all ways each component can fail, the effect on the system, and the risk priority. The Risk Priority Number (RPN) = Severity × Occurrence × Detection combines three dimensions into a prioritization score. Components with high RPN receive design changes or additional testing. FMEA is mandated in medical device design (ISO 13485), automotive engineering (IATF 16949), and aviation (DO-178C). The goal is not eliminating all failures—impossible and uneconomical—but ensuring critical failure modes have multiple independent barriers preventing them from causing system failure.

Accelerated Life Testing

Qualifying long-life products (20-year lifetimes for aerospace components) requires accelerated life testing: exposing components to elevated stress (temperature, voltage, vibration) to trigger failures faster, then extrapolating to operating conditions. The Arrhenius model relates thermal acceleration: L(T) = A e^{E_a/k_BT}, where E_a is activation energy and T is temperature. Running tests at 125°C instead of 55°C accelerates failure by a factor of 100–1000. Statistical models (typically Weibull or lognormal) fit to accelerated test failure times, then transform back to operating conditions using the acceleration model. Extrapolation assumptions must be validated carefully.

Software Reliability

Hardware reliability has a well-developed mathematical framework; software reliability is harder. Software doesn't wear out—it fails due to latent bugs triggered by specific input combinations. Software reliability models (NHPP models) treat bug discovery as a nonhomogeneous Poisson process: discovery rate decreases as bugs are found and fixed. The Nelson model and Musa-Okumoto model predict total bug count and time to achieve target reliability. Formal verification—mathematically proving software meets its specification—provides the highest reliability assurance for critical systems (aircraft flight control software, cryptographic protocols) but remains expensive and applicable only to limited code sizes.

Conclusion

Reliability engineering translates physics of failure into quantitative probability statements, enabling systematic design for dependability rather than hoping for the best. From the Weibull distribution capturing component lifetimes to fault trees analyzing system failure logic to accelerated testing extrapolating from weeks to decades, the mathematical tools of reliability engineering underlie the extraordinary dependability of modern technology. The discipline's central insight—that reliability is designed and quantified, not assumed—is what makes billion-mile space missions, decades-long implanted medical devices, and trillion-transistor chips feasible.
